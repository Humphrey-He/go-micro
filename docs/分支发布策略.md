# 分支发布策略

本文档说明当前项目的 dev / main / prod 分支治理方式。

## 1. 分支定位

| 分支 | 定位 | 合并方式 |
|---|---|---|
| `dev` | 日常开发集成分支 | 开发提交进入 dev |
| `main` | 已通过 CI 的主干分支 | `dev` 通过 GitHub Actions 自动合入 |
| `prod` | 生产候选/生产稳定分支 | 手动触发 Actions 从 main 晋级 |

## 2. dev 自动合并 main

工作流：

- `.github/workflows/ci.yml`
- `.github/workflows/auto-merge-dev.yml`

流程：

1. 推送代码到 `dev`。
2. `ci` 自动运行质量检查、单测、race、构建和 Docker build。
3. `ci` 成功后，`auto-merge-dev` 自动把 `dev` 合并到 `main`。
4. 合并到 `main` 后，`ci` 会再次在 `main` 上运行。

注意：

- 如果 `dev` 与 `main` 有冲突，自动合并会失败，需要人工处理。
- 如果仓库开启了 main 分支保护且禁止 GitHub Actions 直接 push，需要改成 PR + auto-merge 或配置允许 `GITHUB_TOKEN` 写入。

## 3. main 晋级 prod

工作流：

- `.github/workflows/promote-prod.yml`

流程：

1. 在 GitHub Actions 页面手动运行 `promote-prod`。
2. 默认选择 `main` 作为晋级来源。
3. Actions 重新执行 gofmt、go vet、`go test ./... -race`。
4. 验证通过后，使用 fast-forward 方式推进 `prod`。

原则：

- `prod` 不接收日常开发提交。
- `prod` 只从已验证的 `main` 晋级。
- `prod` 分支代表成熟、可部署、可回滚追踪的迭代版本。

## 4. 推送分支

本地首次创建分支后，需要推送到远端：

```bash
git push -u origin dev
git push -u origin prod
```

推送后 GitHub Actions 才能监听这些分支。
