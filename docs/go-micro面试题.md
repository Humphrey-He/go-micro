# go-micro 面试题（项目实战口径）

本文档基于当前项目实现与工程实践整理，面向面试深度追问的问答稿。回答强调设计取舍、可靠性和工程化细节，避免“只描述功能”。

---

## 1. 微服务设计与服务拆分

**Q: 为什么选择按领域拆分？不是按技术层拆吗？**  
A: 选择领域拆分是为了让服务拥有清晰的业务边界与单一职责。订单服务只负责订单生命周期与状态机，库存服务负责预占/释放/确认，任务服务负责异步编排与重试。这样拆的好处是：
- **职责清晰**：每个服务对外暴露的接口语义一致，容易治理。  
- **可独立扩容**：订单和库存的 QPS 通常高于任务编排，可单独水平扩展。  
- **变更风险可控**：业务规则变更时只影响对应服务，减少跨服务耦合。  
按技术层拆（如“数据库层”“缓存层”）会导致每个服务的业务语义不完整，工程协作复杂且调用链变长。

**Q: 你如何保证服务之间解耦？**  
A: 我做了三层解耦：  
1. **接口层解耦**：内部调用统一用 gRPC 协议，强类型约束与版本控制。  
2. **事件解耦**：订单创建后写入 Outbox，再异步发布到 MQ，任务服务通过事件消费，不依赖订单服务的同步响应。  
3. **数据解耦**：每个服务拥有自己的库表，避免共享数据库引起的跨服务锁与耦合。

**Q: 如何考虑高可用与容错？**  
A: 高可用与容错主要通过三类机制实现：  
- **超时与重试**：gRPC 调用有超时控制，任务补偿走重试/死信机制。  
- **最终一致性**：通过 Outbox + MQ 实现异步一致性，不用强一致分布式事务。  
- **幂等保护**：订单创建、库存释放、取消订单均有幂等语义，避免重复执行引发数据错乱。

## 2. gRPC 与消息队列的结合

**Q: 为什么不用全 REST，为什么一定要 gRPC？**  
A: 内部服务调用要求**低延迟、强约束、可治理**。gRPC 的强类型协议能避免“接口漂移”，同时连接复用降低 RTT。对外仍用 HTTP/JSON，兼顾前端生态。

**Q: gRPC 与 MQ 的分工是什么？**  
A:  
- **gRPC**：同步强一致的业务步骤（订单 -> 库存预占）。  
- **MQ**：异步解耦步骤（订单事件 -> 任务编排），降低主链路耗时并隔离下游波动。

**Q: RabbitMQ 消息可靠性如何保证？**  
A:  
1. **Outbox 机制**：订单状态更新和消息写入同事务，防止“订单成功但消息丢失”。  
2. **手动 ACK + DLQ**：消费失败可重试，超过阈值进入死信队列。  
3. **消费幂等**：任务创建使用唯一键约束，重复消息不会生成重复任务。

## 3. 状态机与任务编排

**Q: 订单/任务状态机如何设计？**  
A:  
- 订单状态：`CREATED -> RESERVED -> PROCESSING -> SUCCESS`，允许 `CREATED/RESERVED -> CANCELED`。  
- 任务状态：`PENDING -> RUNNING -> SUCCESS/FAILED/DEAD`。  
状态机用于控制状态跃迁，避免并发更新导致的“乱跳”。

**Q: 如何保证幂等？**  
A:  
1. **订单创建幂等**：request_id 唯一索引 + 查询回显。  
2. **任务创建幂等**：订单号 + 任务类型唯一约束。  
3. **库存释放幂等**：释放接口检查状态，已释放直接返回成功。

**Q: 任务失败如何补偿？**  
A:  
失败任务进入重试队列（指数退避），超过最大次数进入 DEAD。对于超时取消链路，失败会再次延迟，最终进入 DEAD，避免无限重试。

**Q: 最终一致性怎么保证？**  
A:  
同步链路保证订单与库存一致，异步链路通过 MQ + 重试保证任务最终可达。补偿逻辑（超时取消与库存释放）保证最终不会残留“占用库存”。

## 4. 超时取消与补偿机制

**Q: 超时取消如何触发？**  
A:  
订单 RESERVED 后，会创建一条 15 分钟延迟的 TIMEOUT_CANCEL 任务。任务触发时检查订单状态，未完成则取消并释放库存。

**Q: 超时取消与库存释放的竞争条件如何处理？**  
A:  
1. 订单取消具有幂等语义，重复调用不会重复变更。  
2. 库存释放支持 `order_id` 幂等释放，已释放直接返回成功。  
3. 超时任务执行时先判断订单状态，已 SUCCESS/CANCELED 直接返回成功。

**Q: 补偿机制如何设计？**  
A:  
补偿逻辑由任务服务触发，利用状态机和幂等释放保证一致性。失败会重试，最终进入死信，避免无限重放。

## 5. 测试设计与工程化

**Q: 测试覆盖重点是什么？**  
A:  
- **状态映射逻辑**：覆盖所有 view_status 分支。  
- **库存幂等释放**：首次释放、重复释放、预占不存在。  
- **timeout cancel 链路**：订单超时取消 + 库存释放 + 重复执行。

**Q: 如何发现并发问题？**  
A:  
使用 `go test -race` 检测竞态。项目包含 worker/retry/timeout，属于典型并发场景，race 能提前暴露共享状态问题。

---

## 6. CI/CD 与自动化测试

**Q: CI 里为什么要跑 -race？**  
A:  
线上并发问题比功能 bug 更难定位，race 能在进入主分支前暴露数据竞争，降低线上风险。

**Q: 如何保证 CI 稳定性与速度？**  
A:  
1. 只跑必要的 `go test` 和 `-race`。  
2. 依赖最小化，避免引入重型外部服务。  
3. 将外部系统模拟为 mock/miniredis/sqlmock，提高可重复性。

## 7. 文档与可观测性

**Q: 为什么重视文档？**  
A:  
文档不仅是交付材料，更是对架构设计的“硬证据”。它能让团队对齐对接口、状态、补偿机制的理解，减少口头沟通成本。

**Q: 可观测性如何设计？**  
A:  
- `/metrics` 暴露 Prometheus 指标  
- `/healthz` `/readyz` 用于健康探针  
- Zap 结构化日志  
链路追踪预留 OpenTelemetry 接入点，便于后续扩展。

---

## 8. 云原生与容器化

**Q: 为什么用 Docker + Kubernetes？**  
A:  
Docker 提供一致的运行环境，K8s 提供自动扩缩容、健康检查、滚动更新与服务发现，使系统具备生产部署形态。

**Q: 如何做滚动更新与回滚？**  
A:  
通过 Deployment 滚动更新策略与 readiness/liveness 探针控制发布节奏，出现异常时可快速回滚到稳定版本。
