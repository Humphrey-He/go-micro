# Go-Micro 项目开发计划

> 基于代码现状分析与目标差距，制定以下分阶段开发计划。

---

## 一、现状分析

### 1.1 已完成功能

| 类别 | 完成项 |
|------|--------|
| 服务架构 | 9 个微服务 (gateway/order/inventory/user/task/payment/refund/activity/price) |
| 通信方式 | HTTP (对外) + gRPC (内部) + RabbitMQ (异步) |
| 数据层 | MySQL + Redis，Outbox 模式 |
| 容器化 | docker-compose.yml (本地) + K8s YAML (9 服务) |
| 可观测性 | /healthz /readyz /metrics 端点，Zap 日志 |
| CI | GitHub Actions (go test + race) |
| 单元测试 | gateway/inventory/order/payment/task 有测试 |

### 1.2 待完善项 (优先级排序)

| 优先级 | 问题 | 影响 |
|--------|------|------|
| P1 | 单元测试覆盖率不足 (user/refund/activity/price/task 主逻辑缺测试) | 代码质量风险 |
| P1 | CI/CD 不完整 (缺少 build + push + deploy 流程) | 交付效率 |
| P2 | OpenTelemetry 分布式追踪未落地 | 线上问题排查困难 |
| P2 | Swagger 注释存在 "???" 占位符 | 文档不完整 |
| P2 | 告警规则未定义 | 运维被动 |
| P3 | 集成测试/E2E 缺失 | 回归风险 |
| P3 | Helm Chart 缺失 | 生产部署复杂 |
| P3 | 服务限流/熔断治理不完善 | 稳定性风险 |

## 二、开发计划

### 阶段一：质量保障 (1-2 周)

#### 1.1 补齐单元测试

**目标**: 核心服务测试覆盖率提升至 70%+

**任务**:
- [ ] `user/service_test.go` — 用户创建、认证、缓存逻辑测试
- [ ] `refund/service_test.go` — 退款发起、状态流转、重试逻辑测试
- [ ] `activity/service_test.go` — 秒杀锁、幂等、防重复测试
- [ ] `price/service_test.go` — 价格计算、历史查询测试
- [ ] `task/service_test.go` — 任务创建、执行器、补偿 worker 主流程测试
- [ ] `gateway/service_test.go` — 聚合查询、跨服务编排 mock 测试

**验收标准**: `go test ./... -cover` 显示覆盖率 > 70%

---

#### 1.2 Swagger 文档清理

**任务**:
- [ ] 扫描所有 handler.go 中的 Swagger 注释
- [ ] 替换 `???` 占位符为真实描述
- [ ] 补充 `Param` / `Success` / `Failure` 完整定义
- [ ] 执行 `make swagger-swag` 重新生成

---

### 阶段二：CI/CD 完善 (1 周)

#### 2.1 流水线增强

**当前**: 仅 `go test` + `go race`

**目标**: 完整 Build → Test → Push → Deploy 流程

**任务**:
- [ ] 添加 `build` job — 多平台编译 (linux/amd64, linux/arm64)
- [ ] 添加 `docker-build-push` job — 构建并推送到 registry
- [ ] 添加 `deploy-staging` job — 部署到 staging 环境的 K8s
- [ ] 添加 `deploy-prod` job — 需手动触发 (protection rule)
- [ ] 完善 `ci.yml` 矩阵: 服务分开 build，避免大仓库重复编译

**预期流水线**:
```
push → test → race → build → docker push → deploy(staging)
                                              ↓
                              deploy(prod) 需手动 approval
```

---

### 阶段三：可观测性增强 (1 周)

#### 3.1 分布式追踪

**任务**:
- [ ] 引入 OpenTelemetry SDK 到各服务
- [ ] gateway-api 透传 trace_id 到下游 (HTTP header / gRPC metadata)
- [ ] order-service/outbox 发布消息时携带 trace context
- [ ] task-service 消费时提取 trace context 并关联
- [ ] 配置 Jaeger exporter (或 Tempo/Elastic)
- [ ] 更新 K8s YAML 添加 OTEL 相关环境变量

**验证**: 在 Jaeger UI 能看到完整调用链 (Client → gateway → order → inventory → MQ → task)

---

#### 3.2 告警规则定义

**任务**:
- [ ] 定义 Prometheus alerting rules:
  - 服务 HTTP 错误率 > 1% 持续 5min
  - gRPC 调用失败率 > 1% 持续 5min
  - RabbitMQ 队列积压 > 1000 消息
  - Outbox PENDING 消息 > 500 持续 10min
  - 订单创建 QPS 下降 50% (可能系统异常)
- [ ] 配置 AlertManager webhook 或 Slack 通知

---

### 阶段四：稳定性治理 (1 周)

#### 4.1 限流与熔断完善

**任务**:
- [ ] gateway-api 添加 HTTP 全局限流 (token bucket / 滑动窗口)
- [ ] 关键 gRPC 调用启用熔断 (inventory/预留、order/创建)
- [ ] 使用 `go-micro/pkg/resilience` 封装现有熔断逻辑
- [ ] 压力测试验证熔断效果 (使用 hey/wrk)

---

#### 4.2 支付超时主动扫描

**当前**: payment 有 `ListTimeoutPayments` 但可能未定期执行

**任务**:
- [ ] payment-service 添加定时扫描器 (每分钟执行)
- [ ] 超时支付 (>15min CREATED 未变化) 自动调用 `MarkTimeout`
- [ ] 验证超时补偿链路 (超时 → 取消订单 → 释放库存)

---

### 阶段五：测试体系 (1 周)

#### 5.1 集成测试

**任务**:
- [ ] 搭建集成测试环境 (使用 docker-compose + testcontainers)
- [ ] 订单创建端到端测试 (gateway → order → inventory → task)
- [ ] 支付失败补偿测试 (创建 → 预占 → 支付失败 → 取消 → 库存释放)
- [ ] 超时取消测试 (模拟 15 分钟超时场景)
- [ ] 退款重试测试 (refund → MQ → 失败重试 → 死信)

---

#### 5.2 E2E 测试 (可选)

**任务**:
- [ ] 使用 `playwright` 或 `cypress` 编写 UI E2E
- [ ] 核心链路: 登录 → 下单 → 支付 → 查询订单
- [ ] 失败链路: 下单 → 库存不足回滚

---

### 阶段六：生产级部署 (1 周)

#### 6.1 Helm Chart

**任务**:
- [ ] 创建 `deploy/helm/go-micro` Chart
- [ ] 支持 values.yaml 配置:
  - 副本数 (replicas)
  - 资源限制 (CPU/Memory requests/limits)
  - 数据库连接串 (外部 secret 引用)
  - 镜像 registry 和 tag
- [ ] 提供 `NOTES.txt` 安装说明
- [ ] 更新 K8s YAML 中的 `image: your-registry/...` 为真实地址

---

#### 6.2 配置中心 (可选)

**如果项目目标为生产级**:
- [ ] 引入 Consul 或 Apollo 配置中心
- [ ] 各服务支持远程配置热更新
- [ ] 敏感配置 (DB密码、MQ密码) 迁移到 K8s Secret

## 四、技术债务 (可选)

| 序号 | 问题 | 建议方案 | 优先级 |
|------|------|----------|--------|
| 1 | 订单与库存非强一致 | 引入 Seata AT 模式或保持现状 (补偿已完整) | 低 |
| 2 | 部分 service 未使用接口抽象 | 重构依赖注入，方便 mock 测试 | 中 |
| 3 | Redis 连接未做连接池配置 | 检查 redis.Options 连接池参数 | 中 |
| 4 | gRPC 未启用 TLS | 生产环境需开启 mTLS | 高 |

## 六、启动建议

1. **优先完成 M1** — 质量是交付底线，测试覆盖不全直接影响后续开发信心
2. **M2 和 M3 可并行** — CI/CD 和可观测性相对独立
3. **M4 在 M3 基础上** — 告警能发现限流问题
4. **M5/M6 可选** — 如果项目目标为演示/面试，M1-M4 已足够
