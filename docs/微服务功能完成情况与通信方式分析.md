# go-micro 微服务功能完成情况与通信方式分析

> 分析范围：`internal/` 下各服务（`activity`、`gateway`、`inventory`、`order`、`payment`、`price`、`refund`、`task`、`user`）及 `proto/` 协议定义。  
> 结论时间：2026-04-07

## 1. 总体结论

- 项目已具备完整的微服务骨架：`HTTP API + gRPC + MQ + MySQL + Redis`。
- `gateway` 作为统一入口，承担鉴权、聚合查询与跨服务编排。
- 核心交易链路（下单、库存预占、任务执行、支付失败补偿、退款补偿）已落地。
- 目前主要待完善点集中在：`payment` 的 gRPC 落地、部分服务接口覆盖面、文档注释完整性。

## 2. 各服务功能完成情况

| 服务 | 核心功能 | 完成度判断 | 说明 |
|---|---|---|---|
| `activity` | 发券、秒杀、活动状态、gRPC | 高 | 秒杀包含 Redis 锁 + 事务 + 幂等保护 |
| `gateway` | 登录鉴权、接口聚合、跨服务编排 | 高 | 已接入 user/order/inventory/task/refund/activity/price，payment 走本地 service 注入 |
| `inventory` | 预占/释放/确认扣减/按订单释放、缓存保护、gRPC | 高 | 具备缓存空值、互斥锁防击穿、预占记录查询 |

| `payment` | 创建支付、状态流转、超时扫描、失败补偿 | 中高 | 核心逻辑完成，但未见 gRPC server/client 落地 |

## 3. 服务间通信方式

### 3.1 HTTP REST（对外）

- 各服务通过 `handler.go` 暴露 HTTP 路由。
- `gateway` 提供统一外部入口，聚合后端服务能力。

### 3.2 gRPC（服务间同步调用）

- `proto/` 已定义 `activity/order/inventory/user/task/refund/price/payment` 协议。
- 已有 gRPC client/server 落地：`activity`、`order`、`inventory`、`user`、`task`、`refund`、`price`。
- `gateway` 在启动时集中创建多个 gRPC 客户端用于编排调用。
- `payment` 虽存在 `proto/payment.proto`，但当前内部目录未见对应 gRPC server/client 代码。

### 3.3 MQ / 异步事件

- `order` 使用 outbox 思路发布事件（如 `order_reserved`）。
- `task` 消费订单事件并触发履约执行。
- `refund` 发起后发布 MQ，由消费者异步执行退款处理与补偿，带失败重试。

### 3.4 数据库与缓存

- 多服务共享 MySQL 基础设施（各服务独立建连接并访问各自表）。
- `user`、`inventory` 等服务使用 Redis 做热点缓存、空值缓存、锁保护。

## 4. 关键调用链（示例）

### 链路 A：下单履约主链路

`gateway -> order(Create) -> inventory(Reserve) -> order(outbox publish) -> task(consume order_reserved) -> task executor -> order(UpdateStatus)`

### 链路 B：支付失败补偿

`gateway/payment API -> payment.MarkFailed -> order.Cancel -> inventory.ReleaseByOrder`

### 链路 C：退款异步补偿

`gateway -> refund.Initiate -> MQ -> refund worker -> refund.Process -> order.Cancel + inventory.ReleaseByOrder`

### 链路 D：任务超时 Saga 补偿

`task timeout worker -> 生成补偿步骤 -> compensation worker -> 调用 order/inventory 完成补偿`

## 5. 待完善项（建议优先级）

1. **高优先级**：补齐 `payment` gRPC server/client，与其他服务通信模式保持一致。
2. **中优先级**：扩展 `task` gRPC 能力（目前能力较聚焦，主要是按订单查询）。
3. **中优先级**：完善 `price` 历史数据写入链路说明或落地接口。
4. **低优先级**：清理部分 Swagger 注释中的占位描述（如 `???`）。
