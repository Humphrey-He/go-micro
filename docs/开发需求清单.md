# Go-Micro 生产化开发需求清单

> 本清单基于当前项目实现、已有开发计划和生产实战差距整理，目标是把项目从“可演示的微服务样板”推进到“可上线、可观测、可回滚、可维护”的生产级工程。
>
> 当前评估：项目核心业务链路和工程骨架已经具备，生产化成熟度约 60% - 70%。下一阶段应优先补齐稳定性、安全、发布、数据迁移、可观测和集成测试能力。

---

## 1. 总体目标

### 1.1 生产化目标

- 支持 dev / staging / prod 多环境部署。
- 核心链路具备完整可观测能力：日志、指标、链路追踪、告警。
- 订单、库存、支付、任务、退款链路在异常场景下可重试、可补偿、可追踪。
- CI/CD 覆盖测试、构建、镜像发布、部署、冒烟验证和回滚。
- 数据库结构具备版本化迁移能力。
- 服务运行在 Kubernetes 中具备基础安全、资源限制和弹性伸缩能力。

### 1.2 当前优势

- 服务拆分较完整：gateway、order、inventory、payment、task、refund、activity、price、user。
- 通信模型完整：HTTP + gRPC + RabbitMQ。
- 已具备 MySQL、Redis、Outbox、补偿、Swagger、Docker、K8s YAML、GitHub Actions 基础。
- 单元测试已有覆盖，全量 `go test ./...` 可通过。

---

## 2. P0 必须完成项

P0 是上生产前的最低门槛，不建议跳过。

### 2.1 配置与密钥治理

**目标**：所有敏感配置和环境差异从代码与静态 YAML 中剥离。

**需求项**：

- [ ] 梳理所有环境变量：MySQL、Redis、RabbitMQ、JWT、gRPC target、HTTP/gRPC 监听地址。
- [ ] 新增 Kubernetes `Secret` 管理数据库、Redis、MQ、JWT 密钥。
- [ ] 新增 Kubernetes `ConfigMap` 管理非敏感配置。
- [ ] 按 dev / staging / prod 拆分配置。
- [ ] 禁止在文档、Dockerfile、K8s YAML 中出现真实密码或生产连接串。
- [ ] 为关键参数提供默认值和启动时校验。

**验收标准**：

- [ ] 本地和 K8s 环境均可通过环境变量启动。
- [ ] Secret 与 ConfigMap 能被 9 个服务正确引用。
- [ ] 缺少关键配置时服务启动失败并输出明确错误。

---

### 2.2 数据库迁移体系

**目标**：从一次性 `schema.sql` 升级为可版本化、可回滚的数据库迁移。

**需求项**：

- [ ] 引入 migration 工具，例如 `golang-migrate`、`goose` 或 Flyway。
- [ ] 将 `deploy/sql/schema.sql` 拆分为版本化 migration 文件。
- [ ] 增加数据库迁移执行脚本或 Makefile 目标。
- [ ] CI 中增加 migration 校验。
- [ ] 为生产变更定义回滚脚本或兼容策略。

**验收标准**：

- [ ] 空数据库可通过 migration 初始化完整结构。
- [ ] 重复执行 migration 不会破坏数据。
- [ ] staging / prod 可按版本升级。

---

### 2.3 Kubernetes 运行时加固

**目标**：让现有 K8s YAML 从“可部署样例”升级为“生产可运行单元”。

**需求项**：

- [ ] 为 9 个服务补齐 `resources.requests` 和 `resources.limits`。
- [ ] 为 9 个服务补齐 `livenessProbe`、`readinessProbe`、必要时增加 `startupProbe`。
- [ ] 增加 `securityContext`，默认非 root 运行。
- [ ] 增加 `PodDisruptionBudget`。
- [ ] 增加 `HorizontalPodAutoscaler`。
- [ ] 增加 `ServiceAccount`，遵循最小权限。
- [ ] 增加 `NetworkPolicy`，限制服务间访问面。
- [ ] 明确 Ingress / Gateway API 入口方案。

**验收标准**：

- [ ] 9 个服务均能在 K8s 中启动、探活、就绪。
- [ ] 滚动发布期间无明显中断。
- [ ] Pod 被驱逐或重建后可自动恢复。

---

### 2.4 CI/CD 发布闭环

**目标**：形成从提交到部署的完整流水线。

**需求项**：

- [x] GitHub Actions 覆盖 gofmt、go vet、go test、race、二进制构建、Docker build。
- [x] main / tag 推送镜像到 GHCR。
- [ ] 增加 staging 自动部署 job。
- [ ] 增加 production 手动审批部署 job。
- [ ] 增加部署后 smoke test。
- [ ] 增加 `kubectl rollout status` 校验。
- [ ] 增加失败回滚策略。
- [ ] 定义镜像 tag 规则：branch、tag、sha。

**验收标准**：

- [ ] PR 阶段阻断格式错误、测试失败、镜像构建失败。
- [ ] main 合并后自动部署 staging。
- [ ] prod 部署必须手动审批。
- [ ] 部署失败时能明确失败阶段并可回滚。

---

## 3. P1 高优先级项

P1 是生产稳定运行的核心能力。

### 3.1 OpenTelemetry 链路追踪

**目标**：打通核心业务链路的分布式追踪。

**需求项**：

- [ ] 引入 OpenTelemetry SDK。
- [ ] gateway HTTP 请求生成或透传 `trace_id`。
- [ ] gRPC client/server 自动注入和提取 trace context。
- [ ] RabbitMQ 消息发布时携带 trace context。
- [ ] task-service 消费消息时恢复 trace context。
- [ ] 接入 Jaeger、Tempo 或 OpenTelemetry Collector。
- [ ] K8s YAML 增加 OTEL 相关环境变量。

**验收标准**：

- [ ] 能在追踪系统中看到 `gateway -> order -> inventory -> MQ -> task` 完整链路。
- [ ] 日志中能输出 trace id，方便日志与 trace 联动。

---

### 3.2 指标与告警

**目标**：异常发生前可预警，异常发生后可定位。

**需求项**：

- [ ] 为 HTTP 请求增加统一指标：QPS、延迟、状态码、错误码。
- [ ] 为 gRPC 请求增加统一指标：耗时、错误、目标服务。
- [ ] 增加业务指标：
  - 订单创建成功率
  - 库存预占失败率
  - 支付失败率
  - 支付超时数量
  - Outbox pending 数量
  - RabbitMQ 队列积压数量
  - 退款重试次数与死信数量
- [ ] 新增 Prometheus alert rules。
- [ ] 新增 Grafana dashboard。
- [ ] 接入 AlertManager webhook 或消息通知。

**验收标准**：

- [ ] 关键服务有独立 dashboard。
- [ ] HTTP 5xx、gRPC 失败率、MQ 积压、Outbox 堆积均可触发告警。

---

### 3.3 消息可靠性增强

**目标**：RabbitMQ 与 Outbox 链路可重试、可恢复、可审计。

**需求项**：

- [ ] 完善 Outbox publisher 的重试、退避、限速。
- [ ] 增加 Outbox stuck 状态识别与告警。
- [ ] 增加消息体版本号。
- [ ] 增加消费幂等表或业务幂等标记。
- [ ] 明确可重试错误、不可重试错误、毒消息处理策略。
- [ ] 增加 DLQ 运维脚本或管理接口。
- [ ] 为 MQ 消息增加 `request_id`、`trace_id`、`event_id`。

**验收标准**：

- [ ] 重复消息不会造成重复扣库存、重复取消订单或重复退款。
- [ ] 消费失败可进入 DLQ，并能被人工或脚本重放。
- [ ] Outbox pending 堆积可被监控和告警。

---

### 3.4 安全体系

**目标**：补齐认证、鉴权、密钥和通信安全。

**需求项**：

- [ ] JWT 密钥进入 Secret 管理。
- [ ] 增加 JWT 过期、刷新、吊销策略。
- [ ] 网关增加统一鉴权中间件和权限模型。
- [ ] 内部 gRPC 增加服务间鉴权，生产环境评估 mTLS。
- [ ] 统一参数校验与错误响应。
- [ ] 日志脱敏：token、密码、手机号、支付相关字段。
- [ ] 增加全局限流和关键接口防刷。
- [ ] CI 增加依赖漏洞扫描。
- [ ] CI 增加容器镜像安全扫描。

**验收标准**：

- [ ] 未授权请求不能访问受保护接口。
- [ ] 敏感字段不会出现在应用日志中。
- [ ] CI 能发现高危依赖漏洞和镜像漏洞。

---

## 4. P2 中优先级项

P2 用于提高长期可维护性和复杂场景可靠性。

### 4.1 集成测试与 E2E

**目标**：验证服务组合后的真实行为。

**需求项**：

- [ ] 使用 docker-compose 或 testcontainers 启动 MySQL、Redis、RabbitMQ。
- [ ] 增加下单主链路集成测试：
  `login -> inventory -> create order -> query order -> query order view`
- [ ] 增加支付失败补偿测试：
  `create order -> create payment -> mark failed -> cancel order -> release inventory`
- [ ] 增加支付超时补偿测试。
- [ ] 增加退款异步重试测试。
- [ ] 增加重复请求和重复消息幂等测试。

**验收标准**：

- [ ] CI 可运行核心集成测试。
- [ ] 核心异常链路有测试覆盖。

---

### 4.2 状态机与补偿治理

**目标**：让订单、支付、库存、任务状态流转更清晰、更可审计。

**需求项**：

- [ ] 集中定义订单状态合法流转。
- [ ] 集中定义支付状态合法流转。
- [ ] 增加状态变更审计表。
- [ ] 补偿失败进入可重试队列或人工处理队列。
- [ ] 明确 Order、Payment、Inventory、Task 的事实源边界。
- [ ] 为状态变更增加 request id / operator / reason。

**验收标准**：

- [ ] 非法状态流转会被拒绝并记录原因。
- [ ] 每次关键状态变更都可追溯。
- [ ] 补偿失败不会静默丢失。

---

### 4.3 Proto 与代码生成规范

**目标**：统一 API / RPC 生成规则，避免生成代码漂移。

**需求项**：

- [ ] 明确生成代码是否提交仓库。
- [ ] CI 校验 proto 生成结果是否最新。
- [ ] 引入 `buf` 或等价工具进行 proto lint。
- [ ] 增加 breaking change 检查。
- [ ] 明确 Gin + swag 与 goctl api 的路线：
  - 若继续 Gin + swag，则 goctl api 仅作为可选生成能力。
  - 若切换 goctl api，则补齐 `api/gateway.api`。
- [ ] 文档中说明 Makefile 生成命令的使用边界。

**验收标准**：

- [ ] proto 变更不会破坏已有客户端而无提示。
- [ ] CI 能发现未同步的生成代码。

---

### 4.4 Helm 或 Kustomize

**目标**：降低多环境 Kubernetes 部署维护成本。

**需求项**：

- [ ] 创建 `deploy/helm/go-micro` 或 `deploy/kustomize`。
- [ ] 支持镜像 registry、tag、replicas、resources、env、secretRef 配置。
- [ ] 支持 dev / staging / prod values。
- [ ] 增加部署说明和回滚说明。

**验收标准**：

- [ ] 单条命令可部署 staging。
- [ ] prod 与 staging 能复用同一模板，仅 values 不同。

---

## 5. P3 优化项

P3 用于提升工程体验、运维效率和长期演进能力。

### 5.1 性能压测与容量评估

**需求项**：

- [ ] 使用 hey、wrk、k6 或 Locust 编写压测脚本。
- [ ] 压测 gateway 下单接口。
- [ ] 压测库存预占并发场景。
- [ ] 压测支付状态更新场景。
- [ ] 输出 QPS、P95、P99、错误率、CPU、内存、数据库连接数。

**验收标准**：

- [ ] 形成容量评估报告。
- [ ] 明确单副本和多副本承载能力。

---

### 5.2 故障演练

**需求项**：

- [ ] 模拟 MySQL 不可用。
- [ ] 模拟 Redis 不可用。
- [ ] 模拟 RabbitMQ 不可用。
- [ ] 模拟 inventory-service 超时。
- [ ] 模拟重复消息和乱序消息。
- [ ] 输出故障演练记录和恢复流程。

**验收标准**：

- [ ] 核心故障场景有明确恢复步骤。
- [ ] 故障期间不会产生不可恢复的数据不一致。

---

### 5.3 运维工具

**需求项**：

- [ ] 增加 Outbox 重放脚本。
- [ ] 增加 DLQ 重放脚本。
- [ ] 增加订单状态修复脚本。
- [ ] 增加库存预占记录检查脚本。
- [ ] 增加支付超时扫描手动触发命令。

**验收标准**：

- [ ] 常见线上数据修复动作有工具支持。
- [ ] 运维脚本有 dry-run 模式。

---

## 6. 建议实施路线

### 阶段 1：生产最低门槛

- 配置与 Secret / ConfigMap
- 数据库 migration
- K8s resources / probes / securityContext
- CI/CD staging 部署

### 阶段 2：稳定性与可观测

- OpenTelemetry
- Prometheus 指标
- Grafana dashboard
- AlertManager 告警
- MQ / Outbox 可靠性增强

### 阶段 3：安全与测试

- JWT 与服务间鉴权
- 日志脱敏
- 依赖和镜像扫描
- 集成测试和核心 E2E

### 阶段 4：工程化与运维

- Helm / Kustomize
- 状态机审计
- 故障演练
- 运维脚本
- 性能压测与容量报告

## 7. 里程碑

| 里程碑 | 内容 | 交付物 |
|---|---|---|
| M1 | 生产配置与 K8s 加固 | Secret、ConfigMap、resources、probe、securityContext |
| M2 | 发布闭环 | staging 自动部署、prod 手动审批、rollout 校验 |
| M3 | 可观测体系 | trace、metrics、dashboard、alert rules |
| M4 | 可靠性增强 | Outbox/MQ 幂等、重试、DLQ 重放 |
| M5 | 测试体系 | 集成测试、E2E、故障注入 |
| M6 | 运维与容量 | Helm/Kustomize、压测报告、运维脚本 |

---

## 8. 当前下一步建议

建议优先从 **P0 - Kubernetes 运行时加固** 开始：

1. 为 9 个服务补齐 `resources`。
2. 为 9 个服务补齐 `livenessProbe` / `readinessProbe`。
3. 抽离 Secret / ConfigMap。
4. 在 GitHub Actions 中加入 staging deploy。

这一步投入适中，收益最大，能最快把项目从“可运行”推进到“具备生产部署雏形”。
