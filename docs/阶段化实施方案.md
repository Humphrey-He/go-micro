# 阶段化实施方案（单体 -> 微服务）

## 第一阶段：单体版本（跑通业务）
目标：先把业务跑通，验证表结构与主流程。
- 功能：登录/用户、订单创建、库存预占、任务记录、订单查询
- 说明：单体内分模块，先形成完整链路

## 第二阶段：拆成微服务
目标：按领域边界拆分并完成服务间通信。
- 拆分：gateway-api、order-service、inventory-service、task-service、user-service
- 技术：gRPC 内部通信、HTTP 对外
- 数据：各服务独立表结构

## 第三阶段：异步与一致性
目标：完善消息驱动与最终一致性。
- Outbox：订单写入与消息写入同事务
- MQ：发布 order.created，task-service 消费
- 补偿：任务失败触发库存释放

## 第四阶段：稳定性与可观测性
目标：提升可靠性与排障效率。
- 统一错误码、幂等控制
- 缓存穿透/雪崩治理
- 指标/日志/链路追踪

## 第五阶段：云原生部署
目标：具备生产级可运行形态。
- /healthz /readyz /metrics
- Docker 多阶段构建
- K8s Deployment + Service + Probe
